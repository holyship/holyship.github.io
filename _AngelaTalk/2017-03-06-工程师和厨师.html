--- title: 工程师和厨师 author: 朱赟 date: '2017-03-06 00:00:00 +0000' --- {% raw %}

在我还没有出国前，因为家庭原因，近距离接触了一些跟酒店相关的事情。尤其是大小饭店厨房里面的流程体系和角色分工。


比如一家二十左右员工的小饭店，厨师一共也就三四个。每个人都要管一大摊子的事：定菜单，采办、配菜、切菜、烧炒煮蒸拌……再比如三四百号员工的大饭店，一整套厨师班子至少也有二三十个人的样子，主厨、大厨、二厨、红案、白案、面点、凉菜……每道菜，也都有负责的人。


偶尔想起，觉得情况和大公司小公司的程序员倒是很类似。小公司的时候，我们从配置工作环境，到开发，到部署，统统都是自己来，甚至偶尔产品和设计的事也沾着点儿边。等到大公司的时候，很少有前前后后一把揽下的情况：开发环境和工具有人负责，实际发布和部署有人负责，然后实际的开发，也是分门别类，支付、安全、搜索、前端、后端、数据库等等等等。


在朋友和同事里，我的两次职场经历，都是加入的时候大概百来号工程师，然后迅速地以每年都翻倍的速度增长着。还有很多，是从微软这样的大公司一步步跳到小公司，负责一个越来越大的摊子。也有不多的一些，是从十几二十个人的小公司加入。


有时候自己脑补着，把每个工程师都看成一个厨师。


有的很像小饭店里的大厨子。什么菜都会做，可以很快的作出一整桌一整桌的菜。偶尔也有几个特别拿手的招牌菜，但是更大的特点是利索、灵活。


有的很像大饭店里专门负责或凉菜或面点或炖菜的师傅。可以在这个饭店做出一手绝活，但是你真让他自立门户，可能又会有抓瞎的可能。


有的习惯单打独斗，个人作战能力极强，但是却不善于合作。他在一个团队里，甚至偶尔还能影响别的厨师的士气和水准。还有些人，就是团队的凝聚力，你看着他好像只是在这几个厨师的搭配上做些调整，那几个菜的配料上做点加减，从不直接做出成品。却因为他的存在，整个厨房的班子的配合更默契，整桌的菜肴变得更协调、更有色香味的平衡。


当然，那些觉得自己是写 Java 还是写 C++ 还是 Ruby 出身的，就很像川菜、湘菜、粤菜的厨子。能把一个菜系的菜做得出神入化，别的菜系的菜虽说凑合也能掌掌勺，但更多的时候把握不好，或者干脆偏见地以为只有粤菜是最高大上的。


如果是刚开始学厨，是加入大饭店对你更有利呢？还是小饭店？如果已经有了一技傍身，长久来说，又是哪里更有前途呢？


小公司的好处，实战经验丰富，做过很多菜，什么菜都要会做。大公司的好处，见过最高档次的宴席前前后后是怎么个流程。需要怎么合作，怎么协调，怎么做到尽善尽美。


小饭店的厨子，你不做也没人做，自然有的是成长的机会。但是如果本身只会做那么几样，一起工作的人也只会那么几道菜，时间长了，你永远都只会做那么几道菜，除非饭店越做越大，有一些新的机会给你。大饭店的厨子，你不做自然有人做，能给你机会负责多样事情的机会也少。很多时候，人际关系、管理代价都很大。但是只要用心，可以学到很多别人的手艺、或是团队合作的。


其实两种经历都有自然是最好的：既见过满汉全席是怎么做的，这样再去小饭店，眼界是有的。也真正自己动手前前后后都负责过，这样才永远不会眼高手低。


和朋友玩笑，以后再跳槽，要去什么样的公司呢？大家都觉得，要么去小公司，做一名大厨，所有东西自己掌勺。要不就去大公司，能负责一摊子事，充分利用自己的能力和可以掌控的人力物力资源，做出更出色的菜品。


所以不论你现在在哪，总有提升的空间。
