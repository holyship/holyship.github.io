随着 ES2015+ 的推出，以及代码转换编译的普及，无论是在代码中或者教程中，很多人都会遇到新的语言特性。这些常常第一次看见时会引起人们摸不清头脑的特性中的一个是 JavaScript 装饰器。

多亏了Angular 2+ ，装饰器已经流行起来了。在Angular里，因为用了Typescript，所以可以使用装饰器。但是在 JavaScript 中，装饰器目前处于 stage 2 提议阶段，这意味着这个特性应该是语言未来要更新的一部分。让我们来看看什么是装饰器，装饰器怎么使用可以让你的代码更加简洁同时更加容易理解。

什么是装饰器?

以装饰器一个最简单的形式来说，一个装饰器仅仅是一个代码片段包装另一个代码片段，字面意思就是“装饰”它。这个概念你之前可能有听说过，如“函数式组合”或者“高阶函数”。 在标准的 JavaScript 中已经有很多这样的用例了，简单的通过调用一个函数来包装另一个函数。

function doSomething(name) {
  console.log('Hello, ' + name);
}

function loggingDecorator(wrapped) {
  return function() {
    console.log('Starting');
    const result = wrapped.apply(this, arguments);
    console.log('Finished');
    return result;
  }
}

const wrapped = loggingDecorator(doSomething);

这个例子产生了一个新的函数，调用变量 wrapped 的结果和 doSomething 执行的结果一样，不同的地方在于被包装的函数调用后，会在原函数调用前后打印日志。

doSomething('Graham');
// Hello, Graham
wrapped('Graham');
// Starting
// Hello, Graham
// Finished

怎么使用 JavaScript 装饰器?

在 JavaScript里，装饰器使用里一个特殊的语法，它的前缀为一个 @ 符号，同时放在被装饰函数的前面。

备注: 在写这篇文章的时候，装饰器目前在 “Stage 2 草案”, 这意味着他们基本已经完成了，但仍然接受变更。

你可以由你所想在同一段代码前放置多个装饰器，它们会被依次按照声明的顺序被执行。

例如:

@log()
@immutable()
class Example {
  @time('demo')
  doSomething() {

  }
}

这里定义里一个类，同时使用里三个装饰器，两个作用于类本身，一个作用于类内部的一个属性：

  • @log 会对所有访问这个类的行为打日志。

  • @immutable 会使这个类变为内部数据成不可变的类型，可能是使用了 Object.freeze 来生成新的实例。

  • @time 会记录每个方法执行的时间，并且会带着一个特殊的标签将这个时间打印出来。

目前，使用装饰器需要有代码转换编译器的支持，因为目前还没有浏览器或者 Node 的发布支持装饰器的版本。 如果你使用 Babel，你可以使用 transform-decorators-legacy 插件来很方便的启用装饰器。

注意: 插件里 “legacy（遗产）” 这个词的使用是因为它支持 Babel 5 的处理方式，但这种方式可能与最终的标准化的方式不同。

为什么使用装饰器?

虽然函数式编程在 JavaScript 里是可行的，但它对不同的代码段（比如类和类的属性）使用相同的技术显然是困难的，甚至是不可能的。 装饰器添加了对类和属性的支持，可以解决这些问题。未来的 JavaScript 版本可能会对一些麻烦难处理的代码添加装饰器的支持。 装饰器也允许在对这些代码的包装器使用简洁的语法，这样可以促进你少写一些抽象的代码，比你的实际想法更加简洁。

不同类型的装饰器

目前，装饰器所支持的唯一类型是对类和类成员的支持。这包括属性，方法，getters 和 settings. 装饰器其实仅仅是一个返回值是函数的函数，同时项目里相应的内容会被装饰。这些装饰器函数在程序第一次运行的时候就被执行，返回被装饰过的代码。

类成员装饰器

属性装饰器被用来应用到一个简单的类成员上。不论是属性，方法，getters 或者 setters。装饰器函数调用需要传入三个参数

  • target – 这个类成员。

  • name – 类成员的名字。

  • descriptor – 成员描述。 这个就是要传给 Object.defineProperty 的对象。

这里使用一个经典的例子 @readonly。实现的方式十分简单，如：

function readonly(target, name, descriptor) {
  descriptor.writable = false;
  return descriptor;
}

从字面上看就是更新了对象描述的 writable 属性，将其设置为 false。

使用在一个类属性上面的例子如下：

class Example {
  a() {}
  @readonly
  b() {}
}

const e = new Example();
e.a = 1;
e.b = 2;
// TypeError: Cannot assign to read only property 'b' of object '#<Example>'

但是我们可以做的更好。实际上，我们可以对不同的行为更换不同装饰函数。例如，我们来对所有的输入输出记录日志。

function log(target, name, descriptor) {
  const original = descriptor.value;
  if (typeof original === 'function') {
    descriptor.value = function(...args) {
      console.log(`Arguments: ${args}`);
      try {
        const result = original.apply(this, args);
        console.log(`Result: ${result}`);
        return result;
      } catch (e) {
        console.log(`Error: ${e}`);
        throw e;
      }
    }
  }
  return descriptor;
}

这种方式用一个新的函数替代了原本的方法，新函数首先打印了参数，然后调用了原本的方法，最后打印了输出。

注意到，我们这里使用了 Spread 操作符 来把从所有要传入的参数自动创建为一个数组。比起原来的使用 arguments 值，这样处理更加现代。

我们能看到下面的使用方法:

class Example {
    @log
    sum(a, b) {
        return a + b;
    }
}

const e = new Example();
e.sum(1, 2);
// Arguments: 1,2
// Result: 3

你会发现，我们必须使用一个比较特别的语法来执行被装饰的方法。不过这个语法本身需要用一整篇文章来解释。简而言之，apply函数允许你用当前的作用域 this 和传入的参数来执行原本的函数。

更进一步，我们可以给装饰器传入一些参数。例如，我们可以这样来写 log 装饰器：

function log(name) {
  return function decorator(t, n, descriptor) {
    const original = descriptor.value;
    if (typeof original === 'function') {
      descriptor.value = function(...args) {
        console.log(`Arguments for ${name}: ${args}`);
        try {
          const result = original.apply(this, args);
          console.log(`Result from ${name}: ${result}`);
          return result;
        } catch (e) {
          console.log(`Error from ${name}: ${e}`);
          throw e;
        }
      }
    }
    return descriptor;
  };
}

现在变的越来越复杂了，但是，我们把它分解成我们已知的：

  • 一个函数, log, 需要一个参数：name

  • 这个函数返回一个函数，而这个函数本身也是一个装饰器

这与之前的 log 装饰器完全一样，除了把 name 属性放在外面使用。

这下面就是是最后的使用方法:

class Example {
  @log('some tag')
  sum(a, b) {
    return a + b;
  }
}

const e = new Example();
e.sum(1, 2);
// Arguments for some tag: 1,2
// Result from some tag: 3

我们可以直观的看到，这种方式允许我们通过我们的应用的标签来区别出不同的日志行。 这个可以实现是因为 log('some tag') 函数是由 JavaScript 运行时直接调用，其返回值作为 sum 方法的装饰器。

类装饰器

类的装饰器作用于类的整个声明范围，装饰器函数被调用时只能传入一个参数，这个参数是将要被装饰的类的构造函数。

注意到装饰器作用于类的构造器而不是每个类创建的实例。这意味着如果你想处理这个实例，你需要自己返回一个被包装后的构造器版本。

一般情况下，类装饰器比起类成员的装饰器没那么有用。因为你在类装饰器里能做的所有事都能用一个简单的函数做完全一样的事。做任何操作后你都要返回一个新的构造函数来替代原来的类构造函数。 回到我们的打印日志的例子，我们来写一下记录了构造函数的参数：

function log(Class) {
  return (...args) => {
    console.log(args);
    return new Class(...args);
  };
}

这里我们接受一个类来作为我们的参数，然后返回一个新的函数来作为新的构造器。这个方法简单的打印了将要传入的参数，然后返回一个应用了这些参数的新的类实例。

例如:

@log
class Example {
  constructor(name, age) {
  }
}

const e = new Example('Graham', 34);
// [ 'Graham', 34 ]
console.log(e);
// Example {}

我们可以看到执行我们的 Example 类的构造函数时会打印出提供给构造器的参数日志。同时构建的结果正是我们所需要的 Example 的实例。 给类装饰器传参数也同样与类成员函数的一样：

function log(name) {
  return function decorator(Class) {
    return (...args) => {
      console.log(`Arguments for ${name}: args`);
      return new Class(...args);
    };
  }
}

@log('Demo')
class Example {
  constructor(name, age) {}
}

const e = new Example('Graham', 34);
// Arguments for Demo: args
console.log(e);
// Example {}

真实世界示例

核心装饰器

有一个超赞的库叫做 Core Decorators，它提供了一些十分有用的通用装饰器，你现在就可以直接使用。这个库提供了一些非常有用的通用功能（比如：记录方法调用的时间，废弃警告，确保某个值是只读的）同时用了更加便捷的装饰器语法。

React

React 很好的使用了高阶组件的概念，这个也是一个简单的 React 组件，这个组件的写法是一个内部包装了另一个组件的函数。

这正是一个使用装饰器的理想选择，因为你基本不需要为了改成装饰器而对原本代码做出改动。举个例子，Redux 有一个 connect 函数，是用来连接 React 组件和 Redux 数据源。 一般情况下，你会这样做：

class MyReactComponent extends React.Component {}

export default connect(mapStateToProps, mapDispatchToProps)(MyReactComponent);

然而，由于装饰器的原理，你可以这样来替代：

@connect(mapStateToProps, mapDispatchToProps)
export default class MyReactComponent extends React.Component {}

这样可以获得相同的功能.

MobX

MobX 广泛使用了装饰器。它允许你很容易的标记你的数据为可观察的或者被计算的，同时使你的类变为观察者。

总结

类成员的装饰器在类的内部提供了一个非常好的方式包装代码，这种方式与你可以在类里面写独立的函数非常类似。装饰器提供了一个非常好的方式来写一些简单的帮助函数。这些函数可以在很多地方，以一种很简洁移动的方式被使用。 只有你的想象力才能限制这种如基础设施般功能的使用！
